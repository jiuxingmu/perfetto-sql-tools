## 文档定位
`doc/prd.md` 仅描述产品整体设计，不包含任何具体插件设计、插件字段、插件 SQL 或插件验收细节。  
插件级需求统一放在 `doc/plugin/*.md`。

## 1. 产品背景
Perfetto SQL 功能强，但对大量一线用户不够友好：表结构理解成本高、SQL 编写门槛高、重复查询场景多、结果可视化效率低。  
本产品的目标是把高频分析能力工程化，提供可配置、可复用、可视化的分析体验。

## 2. 产品目标
- 降低 Perfetto SQL 使用门槛
- 提高常见分析任务的执行效率
- 提供统一的查询执行与结果展示体验
- 支持后续能力持续扩展并保持架构稳定

## 3. 目标用户
- 性能分析工程师
- 客户端研发工程师（应用/系统）
- 一线排障与质量保障人员

## 4. 核心能力
### 4.1 数据与查询
- 支持 trace 导入与基础信息识别
- 支持参数化查询执行
- 支持查询结果复现与回看

### 4.2 结果展示
- 统一展示层：统计卡片、图表、表格、原始数据、SQL 预览
- 面向数据密集场景优化可读性与操作效率

### 4.3 扩展与治理
- 分析能力可扩展
- 配置与展示口径可维护
- 保持新增能力对既有能力低影响

## 5. 信息架构
- 导入区：trace 导入、解析状态、基础概览
- 筛选区：查询参数输入与重跑
- 结果区：摘要、图表、明细、SQL/Raw 切换
- 全局区：错误反馈、上下文说明、导出能力

## 6. 交互原则
- 让用户优先通过配置完成分析，而非手写 SQL
- 当前筛选条件、结果范围、口径说明必须清晰可见
- 查询失败信息要可读且可行动
- 展示切换（可视化/SQL/原始数据）路径短、反馈快

## 7. 技术与实现原则
- 分层清晰：数据处理、查询执行、渲染展示解耦
- 低耦合高内聚：避免巨型组件与重复逻辑
- 统一模型：参数、结果、展示映射尽量一致
- 风格稳定：保持分析工具场景下的简洁与专业
- 克制依赖：优先用现有技术栈完成需求

## 8. 非功能要求
- 可维护性：结构清晰、命名语义化、便于扩展
- 稳定性：新增能力不影响既有路径
- 性能：交互响应及时，支持中高密度结果渲染
- 可观测性：错误定位信息完整

## 9. 设计边界
- 本文档不定义任何插件细节
- 插件专属参数、SQL、输出字段、验收标准由 `doc/plugin/*.md` 维护
