# PRD：进程画像总览

## 1. 插件名称
- 进程画像总览

## 2. 背景与问题
当前页面更像“进程信息列表”，只能回答“有哪些进程”，难以回答：
- 哪个进程最值得优先分析？
- 哪个进程 CPU 最高？
- 哪个进程阻塞最重？
- 哪个进程线程异常？
- 哪个进程存在慢帧或卡顿风险？

当前痛点：
- 信息偏原始，缺少分析结论。
- 页面像查询报表，不像性能分析工具。
- 缺少下一步引导，用户看完不知道先看谁。

## 3. 产品目标
将该插件升级为 trace 排查入口页，先定位“谁有问题”，再引导“为什么有问题”。

目标包括：
- 快速识别活跃进程与异常进程。
- 输出进程级风险排序与结论。
- 提供下一步插件分析建议（CPU / 卡顿 / 等待 / 线程）。

## 4. 目标用户与场景
主要用户：
- 性能分析工程师
- Android / 系统 trace 分析人员
- 前端 / App 性能排查人员
- QA / 测试排障人员

典型场景：
- trace 初步排查
- 卡顿 / 掉帧 / 阻塞定位
- 热点进程识别与范围收敛

## 5. 插件定位
- 当前定位：进程信息列表
- 目标定位：进程画像总览

插件职责：
- 输出进程基础画像指标
- 给出风险等级和优先级
- 提供继续分析入口

## 6. 核心问题定义
该插件必须回答：
- 本次 trace 里有哪些活跃进程？
- 哪些进程最忙（CPU）？
- 哪些进程阻塞最多？
- 哪些进程线程数异常？
- 哪些进程优先级最高，应先分析？

## 7. 输入参数
必填：
- `start_ts`
- `end_ts`

可选：
- `process_name`
- `pid`
- `uid`
- `status`
- `top_n`（默认 20）
- `sort_by`：`cpu_time` / `blocked_time` / `slow_frame_count` / `thread_count` / `active_duration` / `risk_score`
- `only_active`（默认 true）
- `show_raw`（默认 false）

## 8. 页面结构设计
### 8.1 顶部 Trace 概览
建议展示：
- 时间范围
- 进程总数 / 活跃进程数
- 高风险进程数

### 8.2 分析结论区（第一屏）
示例：
- 本区间共有 12 个活跃进程，其中 3 个需要重点关注。
- `com.xxx` CPU 占用最高。
- `system_server` 阻塞时间较高。

### 8.3 进程画像图（第二屏）
建议优先图表：
- 条形图：CPU 时间、阻塞时长、风险分
- （可扩展）气泡图：CPU vs 阻塞，气泡大小为线程数

### 8.4 进程明细表（第三屏）
建议字段：
- 进程名 / PID / UID / 状态
- 活跃时长 / CPU 时间 / 阻塞时间 / 线程数
- 风险分 / 风险等级
- 主要等待类型
- 建议下一步分析

### 8.5 进程详情抽屉（可迭代）
建议内容：
- 基本信息
- 热点线程
- 阻塞片段
- 慢帧记录
- 快捷跳转到相关插件

## 9. 核心功能模块
### 9.1 筛选与排序
支持按时间、进程、状态、是否活跃、排序字段筛选。

### 9.2 进程画像计算
核心指标：
- 活跃时长
- CPU 时间
- 阻塞时长
- 线程数
- 等待次数

### 9.3 风险评分
基于 CPU、阻塞、线程数、活跃时长计算 `risk_score`，并映射：
- 正常
- 关注
- 异常
- 高风险

### 9.4 下一步建议
根据主导风险维度输出：
- 看 CPU 占用分析
- 看主线程卡顿分析
- 看等待原因归因分析
- 看线程信息列表

## 10. 交互要求
- 默认按 `risk_score` 降序。
- 图表与表格语义一致。
- 支持导出结果。
- 空状态文案明确可操作。

## 11. 数据输出要求
汇总字段建议：
- `total_process_count`
- `active_process_count`
- `high_risk_process_count`

表格字段建议：
- `process_name` / `pid` / `uid` / `status`
- `active_duration_ms`
- `cpu_time_ms`
- `blocked_time_ms`
- `thread_count`
- `risk_score` / `risk_level`

## 12. 验收标准
功能：
- 能输出进程总览与风险排序
- 能按关键维度排序
- 能引导下一步插件分析
- 能导出结果

体验：
- 用户可在 10 秒内判断优先分析对象
- 页面信息层级为“结论 > 图表 > 明细”
- 页面语义符合“分析工具”定位

## 13. 开发约束
- 保持组件化，避免单文件堆叠逻辑
- 不改整体架构，不引入不必要依赖
- 代码风格与现有项目保持一致
- App 编排层不写业务逻辑
